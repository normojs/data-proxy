# Product

## Register

product

## Users

The primary users are admins, operators, and developers who run New API as an
operations console for model routing, channels, keys, MCP, Bridge clients,
OpenAPI tools, billing, audits, and incident response. They work in repeated
sessions where speed, scanability, trustworthy state, and safe configuration
changes matter more than visual novelty.

## Product Purpose

New API provides a unified control plane for many model providers and local MCP
/ Bridge capabilities. It helps teams route traffic, inspect provider health,
manage access and quota, review tool calls, reconcile billing, and keep
automation infrastructure observable. Success means an operator can understand
system state quickly, act without guessing, and recover from failures with an
audit trail.

## Brand Personality

Quiet, exact, operational. The interface should feel like a serious control
room: calm under load, dense without being cramped, and explicit about risk.
It should not try to entertain the operator or sell the product while they are
working.

## Anti-references

Do not make the product surface look like a marketing dashboard, a decorative
SaaS landing page, or a novelty AI cockpit. Avoid giant hero panels, purple-blue
gradient branding, glassmorphism, card-heavy decoration, mascot-like empty
states, and invented controls where standard tables, tabs, filters, switches,
menus, and detail panels are expected.

## Design Principles

1. Put operational state before ornament.
2. Make risk visible at the point of action.
3. Preserve familiar product affordances.
4. Optimize for scanning, filtering, comparison, and repeated edits.
5. Keep v1 and v2 reversible until the pilot proves itself.

## Accessibility & Inclusion

Target WCAG 2.2 AA for text contrast, keyboard navigation, visible focus, and
reduced-motion behavior. Do not rely on color alone for status. Dense tables and
operator workflows must remain usable at laptop widths and on mobile fallback
layouts.
